# Story SP-4.2: End-to-End Validation

Status: done

## Story

As a platform maintainer,
I want to verify that exported skills actually work when copy-pasted into a clean project,
so that I'm confident teammates will have a good first experience.

## Acceptance Criteria

1. **An automated validation script** exists at `scripts/portability/validate-exports.js` (CommonJS, executable) that runs structural validation against a staging directory produced by `seed-catalog-repo.js`. The script checks every exported skill for:
   - Valid markdown (no unclosed code fences, no broken markdown link syntax `[text](`)
   - Zero tool-specific syntax (`Read tool`, `Edit tool`, etc. — same FORBIDDEN_STRINGS list)
   - Zero broken internal references (markdown links pointing to files that don't exist relative to the skill dir)
   - `instructions.md` starts with a `# ` heading
   - `instructions.md` contains `## You are` (persona section)
   - `instructions.md` contains `## How to proceed` (workflow section)
   - `README.md` contains all 3 platform install sections
   - `README.md` line count <= 80
   Exit 0 if all pass, exit 1 with detailed report if any fail.

2. **Claude Code smoke test checklist** is documented in a `VALIDATION-REPORT.md` file generated by the validation script. The checklist covers 3 skills (Carson, Winston, Murat) with manual verification steps:
   - Copy `instructions.md` into a clean project's `.claude/skills/<name>/` directory
   - Invoke the skill by name in Claude Code
   - Verify the persona introduces itself correctly (name + role)
   - Verify the skill does not reference `_bmad/`, `bmad-init`, or missing files
   - Verify the skill produces a usable output

   The dev agent runs the automated checks; the manual Claude Code tests are documented as a checklist for the user to execute after the story is done. The validation script generates the `VALIDATION-REPORT.md` with `[PENDING]` markers for manual tests.

3. **Copilot markdown compatibility check** is automated: for each `instructions.md`, verify:
   - No XML tags remain (`<workflow>`, `<step>`, `<action>`, etc.) — Copilot may parse these as HTML
   - No frontmatter blocks (`---\n...\n---`) — Copilot doesn't process YAML frontmatter
   - No Claude-specific tool syntax
   - Content is valid standalone markdown (no dangling references to external files)
   These checks are integrated into the validation script (AC #1).

4. **Catalog README walkthrough verification** is documented as a manual checklist in the `VALIDATION-REPORT.md`:
   - Can a reader identify what each intent category means?
   - Can a reader find a skill matching "I want to brainstorm" within 60 seconds?
   - Are the tier badges clear? Does the reader understand "Ready to use" vs "Framework only"?
   - Does the "How to use a skill" section give enough context to install?

5. **A test file at `tests/lib/portability-validate-exports.test.js`** adds at least 4 tests:
   - **Test 1: validation passes on a seed-generated staging dir.** Run `seed-catalog-repo.js --output <tmpdir>`, then run `validate-exports.js --input <tmpdir>`. Assert exit 0.
   - **Test 2: validation catches a planted forbidden string.** Create a tmpdir with one skill containing `Read tool` in its `instructions.md`. Run the validator. Assert exit 1 and output contains the violation.
   - **Test 3: validation catches missing persona section.** Create a tmpdir with one skill whose `instructions.md` lacks `## You are`. Run validator. Assert exit 1.
   - **Test 4: VALIDATION-REPORT.md is generated.** After running the validator on a clean staging dir, assert the report file exists and contains `[PENDING]` markers for manual tests.

6. **The validation script's CLI interface:**
   - `node scripts/portability/validate-exports.js --input <path>` — validates the staging directory
   - `node scripts/portability/validate-exports.js --input <path> --report <path>` — also writes `VALIDATION-REPORT.md` to the report path
   - `node scripts/portability/validate-exports.js --help` — usage
   - Exit 0 on pass, 1 on validation failure, 2 on usage error

## Dev Notes

### Architecture Compliance (CRITICAL)

- **The validation script is read-only on the staging directory.** It reads files and optionally writes ONE report file. It does NOT modify any exported skill files.
- **CommonJS.** Match existing convention.
- **No new npm dependencies.**
- **Reuses FORBIDDEN_STRINGS** — same 17-entry list as seed-catalog-repo.js and the test files. This is the 6th copy (Epic 3 retro A1 debt continues to grow). If bandwidth allows, extract to a shared module.

### What's automated vs. manual

| Check | Automated | Manual |
|---|---|---|
| Forbidden strings | Yes | No |
| XML tags remaining | Yes | No |
| Persona section present | Yes | No |
| Workflow section present | Yes | No |
| README platforms + line count | Yes | No |
| Markdown validity (code fences, links) | Yes | No |
| Claude Code persona behavior | No | Yes — checklist |
| Copilot markdown interpretation | Structural only | Full test needs Copilot |
| Catalog 60-second discovery | No | Yes — checklist |

The manual checks are documented as `[PENDING]` items in the VALIDATION-REPORT.md. The user fills them in after running the actual tests. This is a realistic approach — you can't automate "invoke Carson in Claude Code and verify he acts like a brainstorming coach."

### Why 3 skills for manual smoke test

Carson (brainstorming), Winston (architect), and Murat (TEA) represent three different persona shapes:
- Carson: CIS agent, rich persona, 8 step files inlined, strategy 2 resolution
- Winston: BMM agent, persona-only (no workflow.md), strategy 1 resolution
- Murat: TEA module agent, strategy 1 resolution, different module

If these 3 work correctly when copy-pasted, the pipeline is trustworthy.

### Previous Story Intelligence (sp-4-1)

- `seed-catalog-repo.js` already has built-in verification (8 checks). The validation script in sp-4-2 is a superset — it adds markdown validity checks, XML tag detection, code fence balance, and the manual test checklists.
- The seed script's verification is a build-time gate (fail → no output). sp-4-2's validator is a QA tool (run after seeding, produce a report).

## Out of Scope

- **Actually running the Claude Code smoke tests** — the story produces the checklist; the user runs the tests manually.
- **Copilot or Cursor end-to-end tests** — would require those tools installed and configured. sp-4-2 only validates markdown structural compatibility.
- **Automated LLM-based validation** (feed instructions to an LLM and check output) — interesting future work but out of scope.
- **CI integration** — future story.
- **Fixing any validation failures found** — if the validator surfaces issues, they're logged and the story is still done. Fixes go into a follow-up story.
