## Successes

1. **Content-only template reuse proved successful** — Story 3.1's 2-describe-block pattern (Activation Sequence + Workflow Content) transferred across all 5 content-only agents (Isla, Mila, Liam, Noah, Max) with agent-specific adaptations only for step counts and content structure. The template works for both template-having (Isla, Max) and template-free (Mila, Liam, Noah) agents.

2. **Parsing utility library matured across stories** — Each story contributed reusable parsing utilities: `extractYamlBlock`, `extractFrontmatterFieldTable`, `extractBodySections` (Story 3.3), `findFinalStepFile`, `extractCompassSection`, `parseCompassTable`, `extractAgentName` (Story 3.4). Each parser was designed from the start to handle real content edge cases.

3. **Bidirectional assertion pattern emerged and propagated** — Story 3.3 discovered forward-only inclusion checks miss actual-side drift. The M1 fix (bidirectional count assertions) was applied proactively in Story 3.4's cross-reference tests. This is the progressive learning pattern at its best.

4. **Tests caught real content drift** — Story 3.4 discovered 3 Emma workflows missing D4-compliant footer blockquotes. No manual review had flagged this. Tests-as-content-validators is proven.

5. **NFR4 case-insensitive consistency became standard** — Story 3.3 caught case-sensitive field checks, Story 3.4 applied case-insensitive patterns from the start. By epic end, NFR4 compliance was the default approach.

6. **Code review finding severity continues downward trend** — Epic p2-1: 1H/11M/9L. Epic p2-2: 0H/9M/5L. **Epic p2-3: 0H/12M/6L.** Third consecutive epic with zero HIGH findings. MEDIUM findings shifted from factual errors to structural improvements (scoped searches, try-catch robustness, footer scope).

7. **Voice consistency validation was a novel capability** — Story 3.2 introduced voice marker validation (signature phrases, vocabulary, low-confidence labeling). Required careful filesystem verification of every marker. Established patterns for testing subjective content properties programmatically.

8. **P0 gate script established** — Story 3.2 added `npm run test:p0:gate` as semantic CI alias. All subsequent stories validated against the gate.

## Challenges

1. **Compass section parsing complexity** — 9 of 22 workflows have secondary tables ("Insufficient Evidence for Routing") within the compass section. Initial parser captured everything, and code review correctly identified that `hasFooter` searched too broadly. Fix: scope search to content before `###` subheadings.

2. **Voice marker verification was labor-intensive** — Story 3.2 required manual verification of every vocabulary word and signature phrase against actual filesystem content. 6 words initially chosen were absent from agent files and had to be replaced. Factual verification of content-level claims remains time-consuming.

3. **Handoff contract schema complexity** — Story 3.3 discovered HC files are markdown documentation with embedded YAML code blocks, NOT YAML-frontmatter documents. Non-standard section annotations (e.g., `(required, 1-3 per artifact)`) required flexible regex patterns. Agent short names (isla, mila) needed mapping to registry IDs.

4. **Lint warnings from unused imports** — Story 3.1 generated 15 unused variable warnings from destructured imports. Fixed by using bare `require()` instead. Shows the need for more disciplined import patterns.

5. **Sprint-status.yaml File List tracking — 5th consecutive epic** — Template-level fix never implemented. Story 3.4 manually included sprint-status.yaml in its File List (progress), but not systemic. Decision: downgrade to team convention, not blocking action item.

## Key Insights

1. **Tests-as-validators for content correctness is proven** — The entire premise of this epic was that automated tests can catch content drift. Story 3.4 finding missing D4 footers, Story 3.2 catching incorrect voice markers, Story 3.3 validating HC schema structure — all demonstrate that P0 tests are an effective content correctness mechanism.

2. **Progressive learning within epics is now systematic** — Story 3.3→3.4 propagation of bidirectional assertions and case-insensitive patterns. Story 3.1→3.2 propagation of STEP_PATTERN extraction. This is no longer ad-hoc — it's how the team works.

3. **Parsing utilities are the foundation for future validation** — The 7+ parsing utilities created in this epic (YAML extraction, frontmatter tables, compass tables, body sections) are reusable infrastructure for any future content validation stories.

4. **Dynamic discovery (NFR5) scales effortlessly** — Every test file in this epic uses `discoverAgents()` with zero hardcoded agent/workflow values. Adding an 8th agent requires zero test code changes. This architecture investment from Epic 2 continues to pay dividends.

5. **Code review as quality escalator, not gatekeeper** — The review process consistently finds structural improvements (scoped searches, robustness, edge cases) rather than blocking bugs. The team ships correct code that reviews make better.
