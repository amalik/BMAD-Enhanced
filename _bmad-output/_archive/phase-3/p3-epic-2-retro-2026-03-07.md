## Successes and Strengths

1. **PSI compounding across the epic** — Story 2.1's code review caught the docs-audit false positive pattern. That lesson propagated into Story 2.2's dev notes, preventing the same mistake in the CHANGELOG v2.0.0 entry. Story 2.2's review caught URL case sensitivity, which Story 2.3 handled flawlessly — zero HIGH or MEDIUM findings. PSI is now confirmed across 4 consecutive epics.

2. **docs-audit as quantitative progress indicator** — Starting at 21 findings and driving to zero gave the team a concrete, measurable progress signal at every step. Story 2.3's final run returning zero findings was the definitive "done" signal.

3. **Bulk replacement efficiency** — sed with find pipes for 22 workflow files, replace_all for consistent patterns. Mechanical work was fast, leaving attention budget for semantic review where it mattered.

4. **Story review step caught discrepancies early** — Story 2.3's initial ref count audit had inaccuracies (agents.md showed 0 refs but had 1; user guides showed 28 but had 33). Parallel Explore agents corrected the counts before implementation, preventing mid-story surprises.

5. **Adversarial code review as reliable safety net** — Story 2.2's HIGH finding (bulk replace destroying the "was" column) would have been invisible to grep verification. The code review caught it.

---

## Challenges and Growth Areas

1. **Bulk `replace_all` is context-blind** — Story 2.2's CHANGELOG had a "was `bmad-install-vortex-agents`" column that was corrupted when the bulk replacement converted the old names in the comparison column to new names, producing nonsensical "was convoke-install-vortex" entries. Similarly, `BMAD-Enhanced` → `Convoke` produced uppercase `C` in GitHub URL and filesystem path contexts.

2. **Create-story ref count inaccuracy** — Initial analysis undercounted references because it relied on quick scans rather than exhaustive grep. Required spinning up 3 parallel Explore agents to get accurate numbers. This is a known limitation of the create-story workflow when dealing with large, distributed reference sets.

3. **Tech debt carry-forward** — `docs-audit.test.js` strict assertions were flagged in Epic 1's retro as "last Epic 2 or first Epic 3 story." Epic 2 completed without addressing it. Two-epic deferral undermines retro credibility.

4. **Missed version reference** — `docs/development.md` line 104 still showed `v1.6.0` for package.json after Story 2.3. The product name was updated but the version in the same file's tree diagram was not. Fixed during this retrospective.

---

## Key Insights

1. **Replacement ordering rule** — When adding new content that references old names (e.g., migration tables with "was `bmad-*`" columns), add the content AFTER bulk replacements, or use a format that won't match replace patterns. This should be a standing rule in dev notes.

2. **Post-replace context audit** — After any `replace_all`, specifically audit URL contexts (must be lowercase) and filesystem path contexts (case-sensitive). Grep verifies presence/absence but not correctness of replacements.

3. **Quality trend as process signal** — The improving quality across stories (MEDIUM → HIGH → clean) demonstrates that PSI works even within a single epic, not just across epics. Each story's review findings became the next story's prevention checklist.
